# Bevia Governance Architecture — Full Spec

> Platform governance → App governance → Tool governance
> Everything uses `@neuroverseos/governance` engine. No reimplementations.
> Every CLI command that's relevant is used. We eat our own cooking.

---

## Governance Hierarchy

```
┌─────────────────────────────────────────────────────┐
│          PLATFORM: bevia-platform.nv-world.md        │
│  Credits, auth, data isolation, audit, rate limits   │
│  Applies to: ALL tools, ALL actions                  │
├─────────────────────────────────────────────────────┤
│    APP WORLDS (tool-specific governance)              │
│                                                      │
│  ┌──────────┐ ┌──────────┐ ┌──────────┐            │
│  │  Unsaid   │ │  Align   │ │  Arena   │            │
│  │  world    │ │  world   │ │  world   │            │
│  └──────────┘ └──────────┘ └──────────┘            │
│  ┌──────────┐ ┌──────────┐ ┌──────────┐            │
│  │  Split   │ │  Tribe   │ │ Reflect  │            │
│  │  world   │ │  world   │ │  world   │            │
│  └──────────┘ └──────────┘ └──────────┘            │
├─────────────────────────────────────────────────────┤
│    USER WORLDS (per-user strategy governance)        │
│                                                      │
│  ┌──────────────────────────────────────────────┐   │
│  │  Align: user's strategy world file            │   │
│  │  (auto-generated from their uploaded docs)    │   │
│  │  Guards = their strategy rules                │   │
│  │  Values = their culture rules                 │   │
│  │  Red lines = their non-negotiables            │   │
│  └──────────────────────────────────────────────┘   │
│  ┌──────────────────────────────────────────────┐   │
│  │  Reflect: user's behavioral world file        │   │
│  │  (auto-generated from conversation analyses)  │   │
│  │  Contact profiles, ego state patterns,        │   │
│  │  archetype alignment, decay config            │   │
│  └──────────────────────────────────────────────┘   │
└─────────────────────────────────────────────────────┘
```

**Evaluation order:** Platform world → App world → User world. Platform rules always win. App rules can tighten but never loosen platform rules. User rules can tighten but never loosen app rules. This is the MentraOS pattern.

## What Each Level Governs

### Platform Level (bevia-platform)
| What | Guards |
|------|--------|
| Credits | Atomic deduct, refund on failure, no negative balance |
| Auth | JWT required, cross-user data blocked |
| Data isolation | No cross-tool leaks, no cross-user access |
| Output framing | No personality judgments, no manipulation language |
| Rate limits | Max AI calls per minute, simulation depth caps |
| Audit | Every action logged immutably |
| Deletion | User data deletion always priority-allowed |

### App Level (per-tool)

**Unsaid world** governs:
- Archetype accuracy claims (never claim "science says Gen Z talks like this")
- Translation honesty (never fabricate what someone "meant")
- Cultural sensitivity (flag potentially offensive archetype stereotypes)

**Align world** governs:
- Strategy document conflicts (detect and surface, never silently merge)
- Org model hierarchy (who can upload, who can check, team permissions)
- Verdict integrity (AI can't override deterministic red line flags)
- Rewrite boundaries (suggest edits, never auto-apply)

**Reflect world** governs:
- Contact profile framing (behavioral observations only)
- Simulation honesty (confidence disclaimers, data requirements)
- Quiet mode (sacred — suppress everything during high intensity)
- Debrief optionality (never forced)
- Archetype contextuality (don't penalize appropriate behavior)

**Arena world** governs:
- Philosophy accuracy (don't misrepresent Stoicism, Buddhism, etc.)
- No prescriptive advice (perspectives, not instructions)
- Multi-lens fairness (no lens is framed as "better")

### User Level (per-user, auto-generated)

**Align user worlds** = their strategy governance rules. Generated by `infer-world` + `configure-world` from their uploaded docs. These ARE world files.

**Reflect user worlds** = their behavioral profile. Contact data, ego state patterns, archetype alignment. Auto-evolves from conversation analyses.

## Align Simulation (your question)

Yes — simulation absolutely works for Align. Here's how:

### Organizational Model Ingestion

When a user uploads docs to Align, we should accept ALL of these:

| Doc Type | What it tells us | How we use it |
|----------|-----------------|---------------|
| **Strategy doc** | Where the org is going | Priorities, guards |
| **Culture doc** | How the org behaves | Values, red lines |
| **Org model** | Who does what at every level | Roles, permissions, decision authority |
| **Team charters** | What each team owns | Sub-models within levels |
| **Process docs** | How work flows | Plan constraints |
| **Competency frameworks** | What "good" looks like per role | Behavioral expectations |

### Org Model → World File

An org model becomes a `# Roles` section in the world file:

```markdown
# Roles

## executive
- name: Executive (C-suite, VP)
- permissions: [approve_strategy, set_red_lines, override_drift]
- constraints: [max_strategy_changes_per_quarter: 2]

## director
- name: Director
- permissions: [propose_strategy_changes, approve_team_charters, check_alignment]
- constraints: [cannot_override_red_lines]

## team_lead
- name: Team Lead
- permissions: [check_alignment, propose_changes, manage_team_charter]
- constraints: [changes_require_director_approval]

## individual_contributor
- name: Individual Contributor
- permissions: [check_alignment, view_verdict]
- constraints: [cannot_modify_strategy, cannot_modify_culture]
```

### Sub-Models (Team-Level Governance)

Some orgs have team-specific rules that exist WITHIN the org model:

```markdown
# Sub-Models

## engineering_team
- parent_role: team_lead
- additional_guards:
  - "All technical proposals require architecture review"
  - "No new dependencies without security scan"
- additional_values:
  - "Code quality over shipping speed" (may differ from org-level "move fast")
- inherits: all org-level values, red lines, priorities
- overrides: none (sub-models can tighten, never loosen)

## sales_team
- parent_role: team_lead
- additional_guards:
  - "All pricing proposals require finance review"
  - "No discounts > 20% without VP approval"
- additional_values:
  - "Revenue targets are hard commitments, not aspirations"
- inherits: all org-level values, red lines, priorities
- overrides: none
```

**The same hierarchy as platform → app → user governance.** Org rules always win. Team rules can tighten but never loosen org rules.

### Align Simulation: "What happens if we adopt this proposal?"

Using the simulate engine:

```
User uploads: "Q3 Hiring Plan — add 20 engineers in 8 weeks"

Align checks it against strategy world:
  ⚠️ DRIFT — strategy says "sustainable growth" (weight 8/10)
  ⚠️ DRIFT — culture says "quality over speed in hiring"
  ✅ ALIGN — priority "expand engineering capacity" (weight 7/10)

User taps: "Simulate impact"

Simulate engine runs the world forward:
  Step 1: 20 hires in 8 weeks → hiring_velocity = "aggressive"
  Step 2: Aggressive hiring → onboarding_quality drops (rule-003)
  Step 3: Onboarding drop → code_quality_risk increases (rule-007)
  Step 4: Quality risk → customer_satisfaction_risk increases (rule-012)
  Step 5: BUT engineering_capacity increases → delivery_speed increases

Result:
  "If you adopt this plan:
   ✅ Engineering capacity increases 40% by Q4
   ⚠️ Onboarding quality likely drops — your culture doc says
      'every hire gets 6-week ramp' but 20 hires in 8 weeks
      means concurrent onboarding at 3x normal rate
   ⚠️ Code quality risk rises — new engineers shipping before
      they understand the codebase contradicts your 'quality gate' guard
   💡 Suggested modification: 12 hires in 8 weeks + 8 hires in
      weeks 9-16. Hits capacity target by Q4 end without
      overwhelming onboarding."
```

## Conflict Resolution UX (your question)

When strategy docs conflict with culture docs:

### Hard Conflicts
```
┌──────────────────────────────────────────┐
│ ⚠️ We found a contradiction              │
├──────────────────────────────────────────┤
│                                          │
│  📄 Strategy Doc says:                   │
│  "Ship weekly. Speed is our advantage."  │
│                                          │
│  📄 Culture Doc says:                    │
│  "No release without full QA sign-off.   │
│   Quality over speed, always."           │
│                                          │
│  These can't both be absolute rules.     │
│  Which takes priority?                   │
│                                          │
│  ┌────────────────────────────────┐      │
│  │ ○ Speed wins — QA is advisory  │      │
│  │ ○ Quality wins — ship when     │      │
│  │   ready, not on a calendar     │      │
│  │ ○ Context-dependent — speed    │      │
│  │   for minor releases, QA for   │      │
│  │   major ones                   │      │
│  │ ○ Let me rewrite these docs    │      │
│  │   to resolve the conflict      │      │
│  └────────────────────────────────┘      │
│                                          │
│           [ Continue →]                  │
└──────────────────────────────────────────┘
```

The user's choice becomes a **priority rule** in the world file:
```json
{
  "id": "speed_vs_quality",
  "label": "Speed vs Quality resolution",
  "description": "When shipping speed conflicts with QA requirements, QA wins for major releases, speed wins for patches",
  "weight": 9,
  "source": "User resolution during onboarding"
}
```

### Soft Tensions
```
┌──────────────────────────────────────────┐
│ 💡 We noticed a tension (not a conflict) │
├──────────────────────────────────────────┤
│                                          │
│  Your strategy emphasizes "innovation    │
│  and risk-taking."                       │
│                                          │
│  Your culture emphasizes "stability      │
│  and predictability."                    │
│                                          │
│  These can coexist — but when a proposal │
│  pushes one at the expense of the other, │
│  we'll flag it as DRIFT and ask you      │
│  which matters more in that context.     │
│                                          │
│  [ Got it — track this tension →]        │
└──────────────────────────────────────────┘
```

Soft tensions become **state variables** that the alignment checker monitors:
```json
{
  "id": "innovation_vs_stability",
  "type": "tension",
  "pole_a": "innovation and risk-taking",
  "pole_b": "stability and predictability",
  "source": "Detected between Strategy Doc and Culture Doc"
}
```

## Migration Plan: Use the Real Engine

### Current (broken)
```typescript
// apps/bevia/functions/shared/governance.ts
// Hand-rolled reimplementation of guard engine
export function evaluateAction(event) { ... } // NOT the real engine
```

### Target (correct)
```typescript
// apps/bevia/functions/shared/governance.ts
import { evaluateGuard, loadWorld } from '@neuroverseos/governance';
import { parseWorldMarkdown } from '@neuroverseos/governance/engine/bootstrap-parser';

// Load platform world + app world at startup
const platformWorld = await loadWorld('./worlds/bevia-platform/');
const appWorld = await loadWorld(`./worlds/${toolName}/`);

// Evaluate through REAL guard engine
export function evaluateAction(event) {
  // Platform rules first (always win)
  const platformVerdict = evaluateGuard(event, platformWorld);
  if (platformVerdict.status === 'BLOCK') return platformVerdict;

  // App rules second (can tighten, not loosen)
  const appVerdict = evaluateGuard(event, appWorld);
  return appVerdict;
}
```

### For Align (user strategy worlds)
```typescript
// User's uploaded strategy becomes a world file via infer-world
import { evaluateGuard } from '@neuroverseos/governance';

// Load user's strategy world from Supabase
const userWorld = JSON.parse(strategy.world_file);

// Evaluate document against user's strategy
const verdict = evaluateGuard({
  intent: 'propose',
  tool: 'document',
  payload: documentText,
}, userWorld);
```

This means Align document checking IS the guard engine. Not a reimplementation. The actual `evaluateGuard()` function evaluating the actual user-generated world file.
